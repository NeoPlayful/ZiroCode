# ZiroCode 大模型中转架构与处理逻辑

## 一、项目定位

ZiroCode 是一个**统一 API 中转平台**。用户通过不同 URL 路径接入不同上游 Provider，无需管理多个 Provider 的 API Key 和计费体系。

**核心能力**：每个 URL 路径前缀（Route）可配置不同的路由模式，支持**单一渠道模式**（主要 Channel + 备用 Channel）和**负载均衡模式**（多 Channel 轮询）。路径决定了请求转发目标，不再依赖请求体中的 `model` 字段做路由。

## 二、路由设计

### 2.1 路由结构（树形组织）

路由按 Path 前缀组织，形成树形结构，支持无限层级扩展：

```
/codex/v1/chat/completions       → [单一渠道] Codex (主) + 备用A
/claude/v1/messages              → [单一渠道] Claude Direct (主) + 备用B
/claude/aws/v1/messages          → [负载均衡] AWS Bedrock 渠道1 + 渠道2 + 渠道3
/claude/gcp/v1/messages          → [负载均衡] GCP Vertex AI 渠道A + 渠道B
/azure/v1/chat/completions       → [单一渠道] Azure OpenAI (主) + 备用C
```

**路由匹配规则**：最长前缀匹配。例如：
- 请求 `/claude/v1/messages` 匹配路由 `/claude/v1`
- 请求 `/claude/aws/v1/messages` 匹配路由 `/claude/aws`
- 不会将 `/claude/aws` 的请求错误匹配到 `/claude/v1`

### 2.2 路由模式

每个路由创建时需选择一种转发模式：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **单一渠道模式** | 绑定一个主要 Channel + 一个备用 Channel，正常走主要，故障切备用 | 单一上游账号，需要容灾保障 |
| **负载均衡模式** | 绑定多个 Channel，请求在它们之间按策略分发 | 多账号负载分摊，或同上游多地部署 |

**一个 Channel 实例可被多个 Route 引用**（例如某个 Channel 同时是 `/codex/v1` 的备用，也是 `/claude/v1` 负载均衡池中的一个渠道）。

#### 2.2.1 单一渠道模式

每个路由绑定两个 Channel：

| 角色 | 说明 | 行为 |
|------|------|------|
| **主要 (Primary)** | 正常情况下的转发目标 | 所有流量默认走主要 Channel |
| **备用 (Backup)** | 主要不可用时的容灾目标 | 仅在主要 Channel 故障时切换 |

**备用切换规则**：

| 触发条件 | 切换行为 | 回切策略 |
|----------|----------|----------|
| 主要 Channel 返回 5xx 错误 | 当前请求 + 后续请求切到备用 | 手动恢复——管理员确认后切回 |
| 主要 Channel 超时（无响应） | 当前请求 + 后续请求切到备用 | 手动恢复 |
| 主要 Channel 完全不可达 | 当前请求 + 后续请求切到备用 | 手动恢复 |

**关键设计**：
- 切换后**不会自动回切**，避免频繁抖动
- 管理员确认主要 Channel 恢复后，**手动执行回切**
- 备用本身也可能挂掉，此时返回 502
- 备用是**独立的 Channel 实例**，不跨路由共享——`/claude/v1` 的备用不会自动将流量转给 `/claude/aws`

#### 2.2.2 负载均衡模式

每个路由绑定一个 Channel 池（多个渠道），请求在渠道之间按策略分发：

| 策略 | 说明 |
|------|------|
| **轮询 (Round Robin)** | 按顺序轮流分发请求 |
| **加权轮询 (Weighted Round Robin)** | 按权重比例分发，权重越高分配越多 |

渠道状态说明：

| 渠道状态 | 行为 |
|----------|------|
| 正常 | 参与负载均衡轮询 |
| 故障（连续 N 次失败） | 自动移出轮询池，进入冷却期 |
| 冷却期结束 | 自动放回池中参与轮询 |

**负载均衡模式没有"主备"概念，所有渠道平等参与分发，故障渠道自动摘除。**

## 四、核心模块详解

### 4.1 API Key 系统

用户通过 Dashboard 的 `/keys` 页面管理 API Key。

| 功能 | 前端页面 | 后端接口 |
|------|----------|----------|
| 列表查询 | KeysPage | `GET /api/keys` |
| 创建 | KeysPage | `POST /api/keys` |
| 删除 | KeysPage | `DELETE /api/keys/:id` |

**Key 属性**：
- `name` — 自定义名称，便于管理
- `key` — 实际的密钥字符串，创建时仅展示一次
- `isActive` — 启用/禁用状态
- `rateLimit` — 每分钟最大请求数
- `maxTokens` — 单次请求最大 tokens
- `lastUsedAt` — 最后使用时间

**认证与配额是所有路由共享的** —— 无论请求哪个 URL 路径，都使用同一套 API Key 认证和配额体系。

### 4.2 路由（Route）与 Provider 管理

#### 4.2.1 Route 数据结构

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `path` | string | URL 路径前缀（最长前缀匹配） | `/claude/v1` |
| `displayName` | string | 显示名称 | `Claude Direct API` |
| `mode` | enum | 路由模式 | `single` / `load_balance` |
| `isActive` | boolean | 是否启用 | `true` |
| `status` | enum | 当前路由状态 | `active` / `failed` / `manual_pause` |

**单一渠道模式特有字段**：

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `primaryChannelId` | string | 主要 Channel ID | `ch_openai_main` |
| `backupChannelId` | string | 备用 Channel ID | `ch_openai_backup` |
| `activeChannel` | enum | 当前在用 | `primary` / `backup` |

**负载均衡模式特有字段**：

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `channelIds` | string[] | 关联的渠道 ID 列表 | `["ch_1","ch_2","ch_3"]` |
| `strategy` | enum | 负载均衡策略 | `round_robin` / `weighted` |
| `fallbackAction` | enum | 全渠道故障时的兜底行为 | `return_502` / `none` |

#### 4.2.2 Channel 数据结构

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `id` | string | 唯一标识 | `ch_openai_main` |
| `name` | string | 内部名称 | `openai-main` |
| `displayName` | string | 显示名称 | `OpenAI 主通道` |
| `baseUrl` | string | 上游 API 基地址 | `https://api.openai.com` |
| `apiKey` | string | 上游 API Key | `sk-xxxxxxxx` |
| `models` | string[] | 该渠道提供的模型列表 | `["gpt-4","gpt-3.5-turbo"]` |
| `weight` | int | 加权轮询权重 | `1` |
| `healthStatus` | enum | 健康状态 | `healthy` / `unhealthy` / `unknown` |
| `isActive` | boolean | 是否启用 | `true` |
| `priority` | int | 优先级 | `0` |

#### 4.2.3 管理功能

| 功能 | 说明 |
|------|------|
| **新增路由** | 配置路径前缀，选择路由模式（单一渠道 / 负载均衡） |
| **单一渠道配置** | 绑定主要 Channel + 备用 Channel |
| **负载均衡配置** | 关联多个 Channel，选择分发策略（轮询 / 加权轮询） |
| **编辑路由** | 修改路径映射或更换关联的 Channel |
| **启用/禁用路由** | 动态控制路由是否参与匹配 |
| **测试 Channel** | 验证 Channel 连通性 |
| **手动回切**（单一渠道） | 主要 Channel 恢复后，手动将流量从备用切回 |
| **查看路由状态** | 展示当前路由模式、关联 Channel 健康状态、当前在用（主要/备用） |

### 4.3 订阅与配额体系

#### 4.3.1 订阅类型

| 类型 | 标识 | 特点 |
|------|------|------|
| 按量付费 | `PAY_AS_YOU_GO` | 固定配额，用完即止，可续费/兑换补充 |
| 月卡 | `MONTHLY` | 月度配额，每月自动重置，含月度上限 |
| 永久 | `PERMANENT` | 一次性购买，长期有效 |

#### 4.3.2 配额管理

- 配额以 Token 为单位计量
- 展示时使用中文单位：亿（100,000,000）、万（10,000）
- 月卡含有两种配额：总配额（长期累计）和月度配额（按月重置）
- 配额在所有路由间**共享消耗**

#### 4.3.3 兑换码系统

管理员可通过 Admin 后台批量生成兑换码，用户进入 `/subscription` 页面输入兑换码增加配额。

| 功能 | 接口 |
|------|------|
| 生成兑换码 | `POST /api/admin/redeem-codes` |
| 兑换 | `POST /api/subscriptions/redeem` |
| 查询计划 | `GET /api/subscriptions/plans` |
| 购买计划 | `POST /api/subscriptions/buy` |
| 支付跳转 | `POST /api/payments/create-checkout` |

### 4.4 用量统计

#### 4.4.1 用户端 (UsagePage)

- **汇总卡片**：总调用次数、总 tokens、配额消耗
- **每日趋势**：按日展示调用量的柱状图
- **模型/路由分布**：按路由路径 + model 统计调用次数与 tokens
- **最近记录**：最近 20 条调用明细（时间、路由路径、模型、实际转发 Provider、tokens、quota、状态码）

#### 4.4.2 管理端 (AnalyticsPage)

- **概览**：总调用、总 tokens、总成本（¥）
- **路由维度分析**：按路由路径聚合统计
- **成本趋势**：ECharts 折线图
- **模型分布**：ECharts 饼图
- **CSV 导出**：`/api/analytics/export?format=csv`

#### 4.4.3 后端接口

| 接口 | 数据内容 |
|------|----------|
| `/api/user/usage` | 总览 + 日趋势 + 路由/模型分布 + 最近记录 |
| `/api/analytics/overview` | 总计数据（调用/tokens/成本） |
| `/api/analytics/cost` | 每日成本时序数据 |
| `/api/analytics/models` | 路由 + 模型维度聚合数据 |

## 五、开发者接口规范

### 认证方式

| 场景 | 方式 |
|------|------|
| Web Dashboard | Cookie (Session) |
| API 调用 | Bearer Token (API Key) |

### 请求方式

不同路由使用不同的 URL 路径，请求体和响应格式由对应的 Provider 决定：

```bash
# 例 1：调用 Codex（OpenAI 兼容格式）
curl https://localhost/codex/v1/chat/completions \
  -H "Authorization: Bearer zk-xxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4",
    "messages": [{"role": "user", "content": "Hello"}]
  }'

# 例 2：调用 Claude Direct API（Claude 原生格式）
curl https://localhost/claude/v1/messages \
  -H "Authorization: Bearer zk-xxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "claude-sonnet-4-6",
    "messages": [{"role": "user", "content": "Hello"}]
  }'
```

**关键点**：请求体/响应体是**原样透传**，不做格式转换。路径决定了转发目标和 Provider，不依赖 body 中的 `model` 字段做路由。

### 错误码定义

| 错误码 | HTTP 状态码 | 说明 |
|--------|-------------|------|
| `UNAUTHORIZED` | 401 | API Key 无效或缺失 |
| `FORBIDDEN` | 403 | 无权限或 Key 被禁用 |
| `NOT_FOUND` | 404 | 路由路径无匹配项 |
| `ROUTE_FAILED` | 502 | 路由对应 Provider 全部不可用（主要 + 备用均故障） |
| `BAD_REQUEST` | 400 | 请求格式错误 |
| `QUOTA_EXHAUSTED` | 403 | 配额不足 |
| `INTERNAL` | 500 | 服务端内部错误 |

## 八、关键技术要点

1. **路径驱动路由**：路由决策基于 URL 路径（最长前缀匹配），而非请求体中的 model 字段，降低用户调用复杂度
2. **双模式支持**：路由创建时可选择**单一渠道模式**（一主一备，手动回切）或**负载均衡模式**（多 Channel 轮询，故障自动摘除），适应不同场景
3. **手动回切，避免抖动**（单一渠道）：故障切换后不会自动回切，由管理员确认主要 Channel 恢复后手动操作
4. **自动摘除，冷却恢复**（负载均衡）：故障渠道自动移出轮询池，冷却期结束后自动放回
5. **原生格式透传**：不对请求/响应体做格式转换，保持各 Provider 的原始 API 协议
6. **树形路径扩展**：路径按树形组织，支持无限层级扩展，新增路由不影响现有路径
7. **前置配额校验**：请求到达 Provider 之前完成配额检查，避免无效上游请求浪费成本
8. **全链路追踪**：从认证 → 路由 → 转发 → 用量记录，每一步都可追溯
9. **配置可管理**：路由→Channel 映射关系通过 Admin 页面动态配置，支持树形展示和管理
